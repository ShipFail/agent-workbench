The crafting table for AI agents

Agent Workbench (AWB) is an MCP-powered workbench where agents can design, ship, evolve, and retire their own tools — backed by a memory-aware inventory that behaves like short-, mid-, and long-term human memory.

Instead of hard-wiring a 50-tool manifest into every agent and praying the router behaves, AWB lets agents synthesize just-in-time tools for the concrete job in front of them, promote what works, and quietly forget the rest. Fewer always-on tools, more compounding behavior.

Stop wiring 50 tools into every agent.

Most "tool-using" agents are configured like overgeared Minecraft players: every pickaxe, every sword, every potion equipped at once. Context windows saturate, routers thrash, and the marginal tool hurts more than it helps.

AWB inverts the pattern. Agents start lean, then strategically craft and equip tools for the job at hand, promoting proven tools into a reusable inventory and archiving the rest. Less action space per step, more depth per token.

Static catalog → dynamic workbench

Replace hard-coded manifests with an MCP workbench where agents manufacture tools on demand, then reuse or discard them based on live feedback, not wishes.

Tools as part of your knowledge graph

Inventory ⇒ tool memory

Equipped ⇒ active toolset

Enchanting ⇒ evolution

Agents can refactor, specialize, and "enchant" tools as tasks recur, converging on stable long-term patterns instead of re-inventing from scratch.

A tiny MCP surface with big emergent behavior.

AWB exposes a deliberately small set of MCP tools. You get clear primitives, not another sprawling SDK.

  • awb.craft

    Given a spec, synthesize a new tool and register it into inventory.

  • awb.call

    Invoke a crafted tool by id. Log success, latency, and usage counts.

  • awb.search

    RAG over your tool inventory: embeddings × usage × recency × tier.

  • awb.list

    Introspect what exists in short-, mid-, and long-term storage.

  • awb.delete / awb.archive

    Let agents prune or retire stale tools instead of hoarding dead weight.

On the ops side, you only ever touch npx awb start and a small config file. Everything else emerges from the feedback loop between your agents and their workbench.

agent.ts Using AWB via MCP 
// 1) Agent decides current task is under-specified
const spec = {
  name: "normalize_csv_schema",
  description: "Normalize heterogeneous CSV columns into a canonical schema.",
  tags: ["etl", "csv", "data-cleaning"],
};

// 2) Craft a dedicated tool for this task via MCP
await mcp.call("awb.craft", { spec });

// 3) Retrieve it using semantic search
const tools = await mcp.call("awb.search", {
  query: "normalize messy csv",
  top_k: 3,
});

// 4) Equip + execute the best candidate
const best = tools[0];
const result = await mcp.call("awb.call", {
  tool_id: best.tool_id,
  params: { csv: inputCsv },
});

Where AWB lives in your stack.

AWB is not another orchestrator. It sits next to your existing agent runtime as a dedicated workbench and tool inventory, exposed via MCP.

Compatible with any MCP-aware agent framework.

1. Your orchestrator

LangGraph, custom planner, homegrown router — anything that can talk MCP and reason about tools.

2. AWB MCP server

A single process started with npx awb start. Exposes craft/call/search/list.

3. Your runtime + storage

Node/Python/HTTP substrate for executing tools, plus whatever vector store and DB you already use in prod.

FAQ

A few sharp questions we've already heard from infra-heavy teams and YC-style founders.

Every new tool increases routing entropy and inflates the action space the model has to reason over. At some point, the marginal tool makes the system strictly worse. AWB keeps the active tool set tiny and shifts everything else into a searchable, decaying inventory instead of the context window.

AWB is intentionally below your orchestrator. Think of it as an MCP-native workbench + inventory layer. Your existing planner / router / graph engine stays in charge; AWB just gives agents a place to craft and manage tools over time.

In practice teams care about: win-rate on complex tasks, latency, cost, and breakage under edge cases. AWB helps by shrinking the decision surface (fewer tools to choose from) while increasing reuse of proven tools. That tends to mean fewer calls, more deterministic behavior, and less prompt spaghetti.

You run npx awb start next to your agent runtime, then add a handful of MCP calls inside your planner: awb.craft, awb.search, awb.call. No new SDK or vendor console to learn.

If you're shipping a single chat UI, don't have an explicit notion of "agent", or aren't instrumenting behavior yet, AWB is probably overkill. It's designed for teams who are already thinking about planners, tool routing, evals, and infra, not for one-off demo apps.

Why not just add more tools?

Because humans don't operate that way either. We keep a tiny, carefully curated active tool belt and dump everything else into the garage. AWB lets your agents do the same.

  • Fewer always-on tools → less routing entropy and better sample efficiency.
  • Heavier reuse of proven patterns → more reliable behavior over time.
  • Inventory as RAG → tools become part of your knowledge graph, not just config.
